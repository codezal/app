Launch a subagent to handle a complex, multi-step subtask autonomously in an isolated context. The agent runs its own tool loop and returns only its final summary.

WHEN TO USE — delegate when a subtask is self-contained and would take you 5+ tool calls:
- Code review: "Review src/auth/validate.ts for null-safety; report severity-tagged findings with line numbers."
- Test writing: "Write unit tests for src/lib/compact.ts covering hysteresis logic; run them and fix failures."
- Debugging: "Diagnose why tests/sessions-store.test.ts fails on the forkAt case; find root cause and fix."
- Refactoring: "Extract the duplicated retry logic in src/lib/providers/ into a shared helper; update all callers."
- Research: "Map how the approval flow works from UI to Rust backend; return a concise architecture summary."
- Doc writing: "Write JSDoc for all exported functions in src/lib/snapshots.ts."

When NOT to use:
- Reading a single known file — use read_file directly.
- Finding a symbol definition — use code_search directly.
- A quick grep for a string — use grep directly.

Usage notes:
- name must be one of the agent types listed in your system prompt catalog.
- task must be DETAILED and SELF-CONTAINED: include file paths, what to change, how to verify, and what to return. The agent cannot see this conversation.
- State clearly whether the agent should write code or only research, and how to verify (e.g. which tests to run).
- For code review of uncommitted changes: you may embed the `git diff` output in the task text, or simply instruct the agent to fetch it — reviewer and refactorer agents have read-only git access (`git diff`, `git log`, `git show`, `git status`, `git blame`).
- The agent's result is not shown to the user — relay a concise summary yourself.
